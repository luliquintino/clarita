# Design: Internacionalização (i18n) — PT, ES, EN

**Date:** 2026-03-29
**Status:** Approved

## Context

Clarita needs to support three languages: Portuguese (default), Spanish, and English. Language preference is stored per user account in the database, synchronized across devices.

---

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| i18n library | `next-intl` | Built for Next.js App Router, type-safe, SSR-friendly |
| Locale in URL | No | App is auth-only — no SEO benefit, adds routing complexity |
| Language storage | `users.language` column (DB) | Per-user, synced across devices |
| Backend translation | Frontend only | Backend error messages stay in PT; only UI is translated |
| User-generated content | Not translated | Clinical notes, goal titles, free-text annotations stay as typed |

---

## Architecture

### Language preference flow

1. **DB**: `language VARCHAR(5) DEFAULT 'pt'` column on `users` table (auto-migration on startup)
2. **Auth response**: `language` returned in login/register/me responses alongside `id`, `role`, `firstName`
3. **localStorage**: stored in `clarita_user` JSON alongside existing fields
4. **Layout**: `layout.tsx` reads `clarita_user.language` → passes to `NextIntlClientProvider`
5. **Change**: Profile page → PATCH `/api/me` → update DB → update `clarita_user` → `window.location.reload()`

No URL prefixes. No middleware routing. Locale is transparent to the URL.

### Translation files

```
dashboard/messages/
  pt.json   ← source of truth (Portuguese)
  es.json
  en.json
```

Organized by namespace:

```json
{
  "auth": { "login": "Entrar", "email": "Endereço de email" },
  "nav": { "patients": "Pacientes", "profile": "Perfil" },
  "journal": { "title": "Check-in de hoje" },
  "symptoms": { "headache": "Dor de cabeça", "anxiety": "Ansiedade" },
  "note_types": { "session": "Sessão", "observation": "Observação" },
  "medication_categories": { "antidepressant": "Antidepressivo" },
  "diagnosis_certainty": { "confirmed": "Confirmado", "suspected": "Suspeito" }
}
```

### Backend data translation strategy

**Known enum values** (fixed set of keys): translated via static entries in JSON files. Component calls `t('symptoms.' + symptom.key)`. If key is missing, `next-intl` returns the key itself as fallback — nothing breaks.

**Free-text user content** (clinical note body, goal titles, annotations): not translated — displayed as typed.

## Language Selector UI

Location: `/profile` page, below personal info section.

```
Idioma / Language / Idioma
[ 🇧🇷 Português ]  [ 🇺🇸 English ]  [ 🇪🇸 Español ]
```

Three buttons with flag emoji + language name. Active language highlighted (green border).

On change:
1. `PATCH /api/me` with `{ language: 'en' }`
2. Update `clarita_user` in localStorage
3. `window.location.reload()`

---

## Verification

1. Login → profile page → switch to English → page reloads in English
2. Logout → login again → page loads in English (preference persisted in DB)
3. Open on different device → logs in → loads in user's saved language
4. Symptom names in ReportSymptomModal appear in selected language
5. Clinical note type labels appear in selected language
6. `npx tsc --noEmit` → zero errors
